## UX Architecture Rules (Apply to Every Feature)

### Rule 1 — One Living Panel, No Chat Spam
- Never call send_message() in response to a button tap
- Always call query.answer() immediately, then edit_message_text() or edit_message_reply_markup()
- The main control panel is a single message that gets edited in-place — never replaced

### Rule 2 — Prefix All Callback Data Strings
Every new callback identifier must use a namespaced prefix, e.g.:
  - v2_users_list
  - v2_invite_view
  - v2_sysinfo
  - v2_notify_sub
This prevents collisions with existing handlers.

### Rule 3 — Back Navigation on Every Sub-Panel
Every sub-menu must have a Back button that returns to the main panel via edit_message_text().
No dead ends. No orphaned states.

### Rule 4 — Destructive Actions Require Confirmation
Any action that stops, restarts, or changes server config must show:
  [Yes, do it]  [Cancel]
before executing. Use the existing confirmation pattern already in bot.py.

### Rule 5 — State Persistence
All mutable state (player sessions, notify subscribers, schedule config, playtime counters)
must be written to state.json using atomic write (write to .tmp then rename).
Never hold critical state only in memory.

### Rule 6 — Access Control
All new handlers must be wrapped with the @restricted decorator (or its async equivalent).
For notify-only tier users, only allow: /status, /players, /notify subscription toggle.
Admins get full access to everything.

### Feature 1 — Invite Code & Password Management

  - Change Password → prompts user to send a new password as a text reply, updates config

### Feature 2 — User Management (Two-Tier Access)

Add "Users" button to main panel.

Two user tiers stored in state.json under "users" key:
  admins: list of Telegram user IDs — full bot access
  notify_only: list of Telegram user IDs — read-only + notify subscription only

Sub-menu:
  - List Users → shows all admins and notify-only users with their Telegram IDs
  - Add Admin → bot asks for a Telegram user ID, adds to admins list
  - Add Notify User → bot asks for a Telegram user ID, adds to notify_only list
  - Remove User → bot asks for a Telegram user ID, removes from whichever tier they are in

The ADMIN_IDS env var remains the root/bootstrap admin list.
Users added via the bot are stored in state.json and merged at runtime.

### Feature 3 — System Info Panel

Add "Sys Info" button to main panel.

When tapped, edit the panel message to show:
  CPU usage: shown as a visual bar, e.g.  [████████░░] 78%
  RAM usage: same bar style
  Disk usage: same bar style
  CPU temp: numeric (read from /sys/class/thermal/thermal_zone0/temp, divide by 1000)
  Container status: Running / Stopped / Unhealthy
  Uptime: human-readable delta from docker inspect State.StartedAt

Bar rendering function (implement once, reuse):
  def make_bar(pct, width=10):
      filled = round(pct / 100 * width)
      return "[" + "█" * filled + "░" * (width - filled) + "] " + str(round(pct)) + "%"

Use psutil for CPU, RAM, disk. If psutil is not installed, add it to requirements.txt.

Add a Refresh button that re-fetches and edits the same message again.

### Feature 4 — Auto Resource Alerts

Background job (JobQueue.run_repeating, interval 60 seconds):

Check CPU and RAM usage every 60 seconds.
If CPU > CPU_ALERT_THRESHOLD (env var, default 85) for 3 consecutive checks:
  Send alert message to all admin chat IDs:
  "Warning: CPU has been above 85% for 3 minutes. Current: X%"
  Reset the consecutive counter after alerting (don't spam).

If RAM > RAM_ALERT_THRESHOLD (env var, default 90):
  Send alert immediately (no consecutive check needed for RAM).
  "Warning: RAM usage at X%. Consider restarting the server."

Add to .env.example:
  CPU_ALERT_THRESHOLD=85
  RAM_ALERT_THRESHOLD=90

### Feature 5 — Session History & Playtime Leaderboard

Player session tracking (extend existing player monitor):

When a player joins: record {player_name, join_time} in state.json under sessions_active
When a player leaves: move entry to sessions_history, compute duration_seconds, add to
  playtime_totals[player_name] (cumulative seconds)

state.json schema additions:
  sessions_active: { "PlayerName": "2026-04-24T01:00:00Z" }
  sessions_history: [ { "name": "X", "joined": "...", "left": "...", "duration_s": 3600 } ]
  playtime_totals: { "PlayerName": 7200 }

New bot commands:
  /history — shows last 20 session entries from sessions_history, formatted as a table:
    PlayerName | Joined | Duration
  /playtime — shows leaderboard sorted by total playtime descending:
    1. PlayerName — 4h 23m
    2. PlayerName — 2h 01m
    ...

Duration formatting helper (implement once):
  def fmt_duration(seconds):
      h = seconds // 3600
      m = (seconds % 3600) // 60
      if h > 0:
          return f"{h}h {m:02d}m"
      return f"{m}m"

### Feature 6 — Idle Auto-Stop

Background job (JobQueue.run_repeating, interval 300 seconds / 5 minutes):

If the server is running AND sessions_active is empty for IDLE_TIMEOUT_MINUTES consecutive
checks (env var, default 60 minutes):

Step 1 (at T-5 minutes): Send warning to all admin IDs:
  "Server has been idle for 55 minutes. Auto-stopping in 5 minutes unless someone joins."

Step 2 (at T+0): If still empty, stop the container and notify:
  "Server auto-stopped after 60 minutes of inactivity. Use /start to bring it back."

Cancel the countdown if a player joins before T+0.

Add to .env.example:
  IDLE_TIMEOUT_MINUTES=60

### Feature 7 — Notify Me (One-Shot Join Subscription)

When a user sends /notify (or taps a Notify Me button):
  If the server is already running: reply "Server is already online!"
  If stopped: register their chat_id in state.json under notify_waitlist

When the server starts (either manually or via schedule):
  After container is confirmed running, send to all waitlist subscribers:
  "The Windrose server is now online! Use your invite code to join."
  Then clear the notify_waitlist.

Available to both admin and notify_only tier users.

### Feature 8 — Scheduled Restart

Add "Schedule" button to main panel (admin only).

Sub-menu:
  - View Schedule → shows current SCHEDULE_RESTART_TIME (e.g. "03:00 UTC") or "Disabled"
  - Set Time → bot prompts for HH:MM UTC, validates format, saves to state.json
  - Enable/Disable toggle → toggles schedule_enabled in state.json

Implementation:
  On bot startup, read schedule_enabled and schedule_time from state.json.
  If enabled, register a JobQueue.run_daily job at the configured time.
  The job: stop container, pull update, restart container, notify admins.
  When the user changes the schedule via bot: cancel existing job and register new one.

Add to .env.example:

  SCHEDULE_ENABLED=false

## ADR-009 — Two-Tier User Access Model

Status: Accepted
Date: 2026-04-24

Context:
The bot currently uses a flat ADMIN_IDS whitelist. Friends who only want
join/leave notifications must be added as full admins, giving them access
to stop/start/restart controls.

Decision:
Introduce a two-tier model:
  Tier 1 (Admin): Full control — start, stop, restart, config, user management
  Tier 2 (Notify-only): Read access + notify subscription only

Admins are stored in ADMIN_IDS (env, bootstrap) plus state.json users.admins.
Notify-only users are stored in state.json users.notify_only.

The @restricted decorator is extended to accept an optional required_tier parameter.

Consequences:
  Positive: Friends can receive notifications without getting server control
  Positive: Owner can delegate admin rights without editing .env
  Negative: Slightly more complex access check logic in the decorator

## ADR-010 — psutil for System Metrics

Status: Accepted
Date: 2026-04-24

Context:
The sys info panel needs CPU, RAM, and disk metrics. Options considered:
  A. Parse /proc/stat and /proc/meminfo manually
  B. Shell out to top/free and parse stdout
  C. Use psutil library

Decision:
Use psutil (option C).

Rationale:
  psutil is the de-facto standard for system metrics in Python
  Single import, cross-platform, no shell parsing fragility
  Adds ~2MB to the venv — negligible on a 24GB RAM instance
  Already likely present if any other monitoring tool is installed

Consequences:
  Add psutil to requirements.txt
  Bot startup fails fast with ImportError if psutil is missing (desired behavior)

## ADR-011 — Idle Auto-Stop via JobQueue Polling

Status: Accepted
Date: 2026-04-24

Context:
The server wastes resources when no players are online.
Options:
  A. Idle auto-stop via polling sessions_active every N minutes
  B. Idle auto-stop via server log event (server emits idle event)
  C. No auto-stop, rely on manual shutdown

Decision:
Option A — polling sessions_active every 5 minutes via JobQueue.run_repeating.

Rationale:
  sessions_active is already maintained by the player monitor
  No dependency on server emitting a specific log event (fragile)
  5-minute polling granularity is acceptable for a gaming server
  The 5-minute pre-warning gives admins time to cancel if needed

Consequences:
  Auto-stop can be off by up to 5 minutes from the exact idle threshold
  Players who join and immediately leave within a 5-minute window reset the counter
  Idle state is persisted in state.json so bot restarts don't reset the timer incorrectly

## ADR-012 — Dynamic JobQueue Scheduling for Restarts

Status: Accepted
Date: 2026-04-24

Context:
Nightly restarts are currently handled by a host-level cron job.
The Telegram bot should be able to manage the schedule dynamically
without requiring SSH access to modify crontab.

Decision:
Move restart scheduling into the bot using JobQueue.run_daily.
The cron job in the host is REMOVED and replaced by this bot-managed schedule.
Schedule config is persisted in state.json so it survives bot restarts.

Rationale:
  Single source of truth for the schedule (bot UI, not crontab)
  Friends/co-admins can adjust the restart time via Telegram
  JobQueue.run_daily supports UTC time targets natively in python-telegram-bot v21

Consequences:
  If the bot process is down at the scheduled time, the restart does not happen
  Mitigation: systemd Restart=on-failure ensures bot is always running
  The host cron healthcheck (every 10 min) continues independently and is unaffected
